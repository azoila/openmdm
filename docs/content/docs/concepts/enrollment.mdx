# Enrollment

Enrollment is the one moment in a device's life where it has no server identity yet. Everything that makes the post-enrollment world secure — device-scoped tokens, heartbeats, command authorization — is bootstrapped from this single HTTP request.

That's why enrollment has its own dedicated cryptographic boundary: a pre-shared HMAC secret baked into the Android agent at build time. This page documents exactly how that boundary works so you can reason about it and audit it.

## The shape of an enrollment request

A device enrolls by POSTing to `/mdm/agent/enroll` with an `EnrollmentRequest`:

```typescript
interface EnrollmentRequest {
  // Device identifiers (at least one required)
  macAddress?: string;
  serialNumber?: string;
  imei?: string;
  androidId?: string;

  // Device info
  model: string;
  manufacturer: string;
  osVersion: string;
  sdkVersion?: number;

  // Enrollment metadata
  method: 'qr' | 'nfc' | 'zero-touch' | 'knox' | 'manual' | 'app-only' | 'adb';
  timestamp: string;    // ISO 8601
  signature: string;    // HMAC-SHA256 hex

  // Optional pre-assignment
  policyId?: string;
  groupId?: string;
}
```

If `enrollment.deviceSecret` is configured on the server (it always should be in production), every request is HMAC-verified before a row hits the database.

## The canonical signing form

The HMAC input is **nine fields joined with a pipe character**:

```text
model | manufacturer | osVersion | serialNumber | imei | macAddress | androidId | method | timestamp
```

Empty optional fields are represented as empty strings, not omitted. The separator is a single `|` with no surrounding whitespace. The signature is computed as:

```text
HMAC_SHA256(deviceSecret, message).hex()
```

This canonical form is **load-bearing**. Changing any of these in either `@openmdm/client` or `@openmdm/core` is a wire break that silently prevents every device from enrolling. The enrollment-signature contract test in `packages/core/tests/enrollment-signature.test.ts` exists to catch any drift — if you see it fail in CI, do not "fix" it by updating one side. Understand why both sides must agree and update both in a single coordinated release.

### In @openmdm/client

The device-side SDK exports a helper:

```typescript
import { generateEnrollmentSignature } from '@openmdm/client';

const signature = await generateEnrollmentSignature(request, timestamp, deviceSecret);
```

### In @openmdm/core

The server-side verification function is also exported so you can use it in custom enrollment flows that bypass the default `/mdm/agent/enroll` route:

```typescript
import { verifyEnrollmentSignature } from '@openmdm/core';

if (!verifyEnrollmentSignature(request, deviceSecret)) {
  throw new Error('Invalid signature');
}
```

## Why HMAC and not a certificate

For Phase 2a (what ships today), HMAC is the right tradeoff:

- **Pro:** No PKI to operate. The secret is a single value you generate with `openssl rand -hex 32` and bake into one file on each side.
- **Pro:** Signature verification is a constant-time operation that works offline.
- **Con:** The secret is symmetric. Anyone who extracts it from an APK and captures the device metadata can forge an enrollment request.
- **Con:** Rotation requires redeploying both sides together.

Phase 2b (roadmap) introduces **hardware-rooted identity** using Android's Keystore attestation. The device generates a keypair in the TEE, submits an attested enrollment request signed with the private key, and the server verifies the attestation chain against Google's roots. At that point, the pre-shared HMAC becomes redundant and will be deprecated.

Until then, **treat the HMAC secret as production-critical**. Store it in your secret manager, not in a git repo, and rotate it if you have reason to believe an APK was extracted.

## The two secrets

New users sometimes assume `enrollment.deviceSecret` and `auth.deviceTokenSecret` are the same thing. They are not, and mixing them up is a security hole.

- **`deviceSecret`** — HMAC secret for enrollment requests. Lives in the APK and the server. Proves "this request is from a device built with our agent."
- **`deviceTokenSecret`** — JWT signing secret for device tokens returned *after* enrollment. Lives only on the server. Proves "this device token was issued by our server."

They serve different purposes:

```
Before enrollment:                       After enrollment:
┌──────────────┐                        ┌──────────────┐
│ EnrollmentRequest │                    │ Bearer token │
│ + HMAC(DEVICE_SECRET)│                 │ signed with JWT_SECRET│
└──────────────┘                        └──────────────┘
         │                                       │
         ▼                                       ▼
   verifyEnrollmentSignature              verifyDeviceToken
```

Rotating `deviceTokenSecret` logs out every device (they get a 401 and refresh their token through re-enrollment or a refresh flow in Phase 2c). Rotating `deviceSecret` requires a new APK build — because the old APK can no longer produce valid enrollment signatures.

## What the server does on a valid enrollment

1. Parse and validate the request body (`EnrollmentRequest`).
2. Run `verifyEnrollmentSignature(request, deviceSecret)`. Reject on false.
3. Run any `enrollment.validate(request)` custom validator you passed in the config. Reject on false.
4. Compute a deterministic `enrollmentId` from the first available identifier: `macAddress → serialNumber → imei → androidId`.
5. Upsert the row in `mdm_devices` with status `enrolled`.
6. Assign the default policy if one exists and no explicit `policyId` was provided.
7. Fire the `device.enrolled` event (your handlers run now).
8. Issue a device JWT signed with `deviceTokenSecret` and return `{ deviceId, token, serverUrl, config }`.

From the agent's perspective, it goes from "I have a pre-shared secret and some device metadata" to "I have a long-lived bearer token" in a single round trip.

## The dev harness

During local development, you usually don't have a real Android agent handy. The dev harness is a script that signs an `EnrollmentRequest` with your local `DEVICE_SECRET` and calls `mdm.enroll()` directly, bypassing the HTTP layer. See [Quick Start §4](/docs/quick-start) for the full example.

## What can go wrong

- **Invalid signature.** The canonical forms diverged between `@openmdm/client` and `@openmdm/core`, or the secret differs between APK and server. The enrollment-signature contract test will catch the code divergence; the secret mismatch you diagnose by inspecting both environments.
- **Timestamp drift.** The current implementation does *not* reject stale timestamps (this is [a known gap](#) tracked for Phase 2b). An attacker who captures a valid enrollment request can replay it — though the replay gives them nothing new because the server upserts by `enrollmentId`.
- **Missing identifier.** If all four of `macAddress`/`serialNumber`/`imei`/`androidId` are empty, the server has nothing to key on and rejects the request.
- **Pre-assigned `policyId`/`groupId` referring to nothing.** The enrollment completes but without the assignment; check the `device.enrolled` event handler to surface this.
